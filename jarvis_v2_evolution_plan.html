Jarvis V2 Evolution

Idea-by-Idea Analysis

Each idea is assessed for achievability, mapped to existing code, and enhanced with concrete improvements.

#1 Remove language detection from wake word + add voice recognition later Achievable

Current state: audio/wake_word.py uses Whisper tiny for Arabic wake word detection — this is a full STT transcription step just to detect a single phrase, which is slow (~300ms) and unreliable.

Assessment: Fully achievable and already partially supported — the code has an WAKE_WORD_AR_ONNX_PATH config for a custom ONNX model that bypasses Whisper. Removing the language detection layer from wake word is a pure win for latency.

Improvement: Instead of just removing language detection, replace the entire Arabic wake word path with a dedicated keyword-spotter ONNX model (ties into idea #3). For voice recognition (speaker ID), add it as a Phase 5 feature using speechbrain ECAPA-TDNN embeddings — small (~25MB), runs on CPU, and can store a voiceprint during setup. This way the wake word fires instantly, and speaker verification runs asynchronously on the captured audio chunk before executing sensitive commands.
#2 Semantic + Hybrid VAD: lightweight local for speech start, complex cloud for turn-taking Achievable

Current state: audio/wake_word.py uses simple energy-threshold VAD (JARVIS_VAD_ENERGY_THRESHOLD=0.015) with fixed silence detection (JARVIS_VAD_SILENCE_SECONDS=0.55). This is fast but misses speech in noisy environments and can't handle conversational turn-taking.

Assessment: Fully achievable. The two-tier approach is the right architecture:

Improvement: Use Silero VAD (1.2MB ONNX, runs at ~2ms per chunk on CPU) as the local tier instead of writing a custom one — it's already proven, bilingual, and handles Arabic speech onset well. For the cloud tier, skip a separate cloud VAD service (adds dependency + latency). Instead, use the existing Whisper/ElevenLabs STT endpoint's built-in endpoint detection as implicit cloud VAD — it already handles turn boundaries during transcription. Add a configurable flush timeout (e.g., 1.5s silence = end of turn) that adapts based on whether the conversation is a command (short timeout) or a question (longer timeout, lets user think). This gives you semantic VAD behavior without a separate service.
#3 Try Picovoice Porcupine for wake word Achievable with caveats

Current state: English wake word uses openWakeWord ONNX (free, open-source, ~2ms inference). Arabic uses Whisper tiny fallback.

Assessment: Porcupine is excellent (sub-10ms, very low false-accept rate) BUT it requires a paid license for custom keywords and has per-device limits on the free tier. For a production project meant to run on any PC, this creates a licensing dependency.

Improvement: Better approach — train a custom openWakeWord model for "جارفيس" using their training pipeline (they provide a Colab notebook). This gives you the same ~2ms inference as the English model, zero licensing cost, and full control. Steps: (1) Generate synthetic Arabic speech samples using edge-tts with different Arabic voices saying "جارفيس", "يا جارفيس", etc. (2) Augment with room impulse responses + noise. (3) Train using openWakeWord's training notebook. (4) Export to ONNX. (5) Ship alongside the English model. Keep Porcupine as an optional premium tier (if user provides their own API key) for environments that demand maximum accuracy.
#4 Slim system prompt builder: minimal and full prompts + remove redundancy Achievable Partially done

Current state: llm/prompt_builder.py already has build_prompt_package() and build_lightweight_prompt(). The weakness plan identified the prompt was 40+ lines — it's been slimmed but can go further.

Assessment: Already partially implemented. The next step is the "remove redundant instructions that the persona already implies" part.

Improvement: Create three prompt tiers instead of two: (1) Micro (3-4 lines + 1 example, for qwen3:0.6b–1.7b) — just role + language + one example. (2) Slim (6-8 lines + 2 examples, for qwen3:4b) — role + language + behavior hints + examples. (3) Full (12-15 lines + 3 examples, for qwen3:8b+) — everything. Auto-select based on get_runtime_model_tier(). Audit every line: if the persona config already says "friendly" or "concise", don't repeat it in the prompt. Add a prompt token counter that logs on first use so you can track regression.
#5 Replace PowerShell for simple operations, use ctypes + Windows API Achievable

Current state: os_control/powershell_bridge.py spawns PowerShell for everything — volume, brightness, lock screen, screenshots. Each spawn costs 150–400ms.

Assessment: Fully achievable and already in the weakness plan. The ctypes approach is proven on Windows.

Improvement: Create os_control/native_ops.py with these specific APIs: volume via IAudioEndpointVolume (COM, not the ancient waveOutSetVolume), brightness via WMI WmiMonitorBrightness (simpler than DDC/CI for laptops), lock via user32.LockWorkStation, sleep via SetSuspendState, screenshot via user32.GetDC + BitBlt. Add a performance decorator that measures ctypes vs PowerShell latency and logs it — this gives you real data. Keep PowerShell as automatic fallback for any ctypes failure. Target: <20ms for volume/brightness vs current 200-400ms.
#6 Refactor post-processing: keep Egyptian rewriter + length control, remove LLM rewrites Achievable Mostly done

Current state: _finalize_success_response() in core/command_router.py already runs only 3 transforms: low-value fallback, Egyptian dialect rewrite, and persona length cap. The old triple-LLM rewrite pipeline was removed.

Assessment: 80% done. The remaining work is to add explicit language enforcement without an LLM call.

Improvement: Add a script-detection language enforcer as the 4th (final) step — no LLM needed. Use Unicode script analysis: if user spoke Arabic and >60% of the response characters are Latin (excluding proper nouns/technical terms), inject a one-line note asking the model to retry in Arabic on the next turn, and cache the failure pattern. This catches the rare case where qwen3 drifts to English mid-response. Also, move the Egyptian colloquial rewriter to the TTS layer (audio/tts.py) so it only runs when text is actually spoken, not when it's displayed in logs.
#7 Improve streaming sentence detection for Arabic Achievable

Current state: llm/ollama_client.py uses a regex-based sentence splitter that works for English punctuation but misses Arabic clause boundaries (Arabic text often has fewer periods and relies on conjunctions like و/ف for flow).

Assessment: Fully achievable. This is the most impactful change for Arabic TTS quality.

Improvement: Implement a dual-mode sentence buffer: (1) Punctuation mode — split on . ! ? ؟ ، ؛ \n (add Arabic semicolon and comma as soft breaks). (2) Fallback flush — if no punctuation found after N=50 characters (not tokens — characters are more predictable for Arabic) or M=12 tokens, look for the last Arabic conjunction (و/ف/ثم) or whitespace boundary and flush there. (3) Hard flush at 100 characters regardless. Add configurable thresholds via env vars: JARVIS_STREAM_AR_SOFT_FLUSH_CHARS=50, JARVIS_STREAM_AR_HARD_FLUSH_CHARS=100. Test with the Egyptian colloquial rewriter to ensure flushed chunks don't break mid-substitution.
#8 Enhance knowledge base: watchdog auto-sync + improved reranking + small reranking model Achievable Partially done

Current state: core/knowledge_base.py already has watchdog support with debounced file system events, polling fallback, FAISS indexing, and hybrid scoring (KB_EMBEDDING_RERANK_WEIGHT + KB_LEXICAL_RERANK_WEIGHT).

Assessment: The foundation is solid. The improvements are about quality of retrieval, not infrastructure.

Improvement: Three layers of improvement: (1) Auto-sync UX — add a configurable notes folder path in .env (e.g., JARVIS_KB_WATCH_FOLDERS=~/Documents/Notes,~/Obsidian) and auto-start watching on boot. Already partially done. (2) Hybrid reranking — add BM25 scoring alongside the current lexical TF-IDF. Use rank-bm25 (pure Python, no deps). Weight: 0.5 embedding + 0.3 BM25 + 0.2 lexical. (3) Small reranking model — defer to Phase 5. Use cross-encoder/ms-marco-MiniLM-L-2-v2 (17MB) as an optional reranker on the top-K candidates. Only load if RAM permits (>12GB). This is a "polish" feature, not critical path.
#9 Better tool prompting: one-line instruction per tool instead of raw dump Achievable

Current state: llm/prompt_builder.py's build_tool_augmented_prompt() injects raw live data results into the prompt. The model sometimes ignores the data or misinterprets it because there's no instruction telling it what to do with each data type.

Assessment: Easy win, high impact on response quality.

Improvement: Create a tool instruction registry — a dict mapping tool names to one-line instructions. Example: {"weather": "Use this weather data to answer. Include temperature and conditions. Say the city name.", "web_search": "Use these search results to answer. Cite the most relevant result. If results are stale, say so.", "kb": "Use this knowledge base context to answer. Prefer KB info over your training data."} Inject the instruction before the data block in the prompt. Also, when injecting live data, add a timestamp so the model knows how fresh it is.
#10 Implement proper VAD-based interrupt while speaking Achievable

Current state: BARGE_IN_INTERRUPT_ON_WAKE exists — if the wake word is detected while TTS is playing, it interrupts. But there's no VAD-based interrupt (user speaks without saying the wake word → Jarvis should stop talking).

Assessment: Achievable but needs careful audio engineering to avoid feedback (Jarvis's own speaker audio triggering the VAD).

Improvement: Two-stage approach: (1) Software echo cancellation — while TTS is playing, subtract the known TTS audio waveform from the mic input (you have the exact audio being played). Use a simple energy-ratio comparison: if mic RMS > (speaker RMS × 1.5), there's likely real speech. (2) Silero VAD confirmation — run the speech chunk through Silero VAD. If both checks pass, fire the interrupt. Add a cooldown (500ms after interrupt before accepting new audio) to prevent bounce. Wire into the TTS playback loop in audio/tts.py and the orchestrator's speaking state.
#11 Let Qwen3 use tool calling for complex intents instead of pure text generation Achievable

Current state: All intents go through the 4-tier cascade (regex → semantic → keyword → LLM text). When the LLM is reached, it generates free text and the router tries to parse it. Complex multi-step requests (e.g., "search for my last report and email it to Ahmed") get lost.

Assessment: Qwen3 supports Ollama's tool/function calling format. This is the right evolution.

Improvement: Don't replace the existing cascade — add tool calling as Tier 3.5 between keyword NLP and pure LLM chat. Define a tool schema for each action family: open_app(name: str), search_files(query: str, location: str), set_timer(minutes: int), web_search(query: str), set_volume(level: int), etc. When the first 3 tiers fail, send the user's text to Qwen3 with the tool definitions. If the model returns a tool call → execute it. If it returns text → treat as chat response. This handles "search for X and open it" as two tool calls in sequence. Use Ollama's /api/chat with tools parameter. Only enable on medium+ hardware tiers (tool calling needs 4b+ to be reliable).
#12 Intent router evolution path: regex → semantic → function calling → pure LLM Achievable Mostly done

Current state: The 4-tier cascade already exists: (1) Regex parser with fast-path, (2) Semantic router (MiniLM embeddings), (3) Keyword NLP (rapidfuzz), (4) LLM fallback. This is almost exactly your proposed evolution.

Assessment: The architecture matches. The gap is the function-calling tier (idea #11) and improving each tier's quality.

Improvement: Rename and formalize the cascade as a 5-tier routing pipeline with clear metrics at each tier: (1) Regex — structural commands, confirmation tokens, file paths (~0ms). (2) Semantic — paraphrase-tolerant intent classification (~5-10ms). (3) Keyword NLP — fuzzy match for noisy STT (~2ms). (4) Function calling — Qwen3 tool use for complex/multi-step intents (~500-1500ms). (5) Pure LLM chat — conversational fallback (~500-2000ms). Add per-tier hit-rate metrics to core/metrics.py so you can see what percentage of requests each tier handles. Target: 70% resolved by tiers 1-3 (instant), 20% by tier 4 (tool calling), 10% by tier 5 (chat).
#13 Scan Windows for all installed apps and add them to app control automatically Achievable

Current state: os_control/app_ops.py has a hardcoded _APP_CATALOG dict with ~20 apps (Notepad, Calculator, Chrome, etc.) and manually written aliases.

Assessment: Fully achievable. Windows has multiple registries and folders where installed apps are listed.

Improvement: Create os_control/app_scanner.py that scans three sources on startup: (1) Registry — HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall and the WOW6432Node equivalent. (2) Start Menu shortcuts — %ProgramData%\Microsoft\Windows\Start Menu + user's Start Menu. (3) App Paths registry — HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths. Parse DisplayName and InstallLocation from each. Auto-generate aliases by splitting the display name (e.g., "Adobe Photoshop 2024" → aliases: ["photoshop", "adobe photoshop"]). Merge with the hardcoded catalog (hardcoded entries win for canonical names + Arabic aliases). Cache the scan result to .jarvis_cache/app_catalog.json with a 24-hour TTL. Re-scan on demand via voice command: "Jarvis, rescan apps".
#14 Confirm with PIN code on power off and restart commands only Achievable Partially done

Current state: core/config.py has JARVIS_SECOND_FACTOR_PIN and JARVIS_SECOND_FACTOR_PASSPHRASE. The confirmation system exists in os_control/confirmation.py with risk tiers. Shutdown/restart are already classified as high-risk.

Assessment: The infrastructure exists. Just need to wire PIN confirmation specifically to power off/restart.

Improvement: Make the PIN flow voice-friendly: when user says "shut down" or "restart", Jarvis responds with "Are you sure? Tell me your PIN to confirm." The user speaks the PIN (digits only), STT transcribes it, and the system compares against the hashed PIN. Add a 3-attempt limit with a 60-second lockout. Also add PIN confirmation for disk format/delete system files if those commands ever get added. Keep simple confirmations (yes/no) for medium-risk actions like closing all apps.
#15 Add mixed language commands (Arabic-English code-switching) Achievable

Current state: JARVIS_STT_MIXED_TREAT_AS_ARABIC=false exists but is off. The command parser has some bilingual patterns but doesn't handle mid-sentence switching well (e.g., "افتح Chrome" or "open الملفات").

Assessment: This is natural Egyptian speech behavior — mixing Arabic verbs with English nouns. Critical for UX.

Improvement: Three changes: (1) STT — set Whisper's language to None (auto-detect) for the main transcription pass instead of forcing AR or EN. ElevenLabs Scribe v2 already handles multilingual. (2) Command parser — add a code-switch normalization step before parsing: detect Arabic verb + English noun patterns (e.g., "افتح Chrome" → extract "Chrome" as app_name, detect "افتح" as open intent). Use the existing app alias lookup on the English portion. (3) Semantic router — the MiniLM model is already multilingual, so mixed-language utterances should route correctly without changes. Test with the 20 most common Egyptian code-switch patterns.
#16 For search/find results, open the folder or search results window Achievable

Current state: os_control/file_ops.py has find_files() that returns results as text. The results are spoken by TTS but the user can't see or interact with them.

Assessment: Fully achievable using Windows Explorer's built-in search and os.startfile().

Improvement: Three result-display modes: (1) Single file found — open the parent folder with the file selected: subprocess.run(["explorer", "/select,", file_path]). (2) Multiple files found — open Explorer with a search query pre-filled: subprocess.run(["explorer", f"search-ms:query={query}&crumb=location:{search_dir}"]). (3) Folder navigation — just open the folder: os.startfile(folder_path). Add a voice confirmation: "I found 3 files. Opening the folder for you." The key is to always give a visual result alongside the voice response.
#17 For settings/timer, open the actual Windows settings page or clock app Achievable

Current state: os_control/settings_ops.py has ms-settings: protocol support. Timer uses threading.Timer + winsound internally.

Assessment: Settings pages already work via ms-settings: URIs. Timer needs to also open the Windows Clock app.

Improvement: (1) Settings — expand the ms-settings: URI map to cover all common settings pages (WiFi, Bluetooth, Display, Sound, Battery, Storage, Apps, Updates, Privacy). When user says "open WiFi settings", launch ms-settings:network-wifi. Add bilingual aliases: "إعدادات الواي فاي" → ms-settings:network-wifi. (2) Timer — dual approach: set a background timer (for the alarm sound) AND open Windows Clock app with a timer pre-configured. Use the ms-clock: protocol or start ms-clock:. On Windows 11, the Clock app supports ms-clock:timer deep link. Fall back to the internal threading.Timer if the Clock app isn't available. This gives the user both a visual countdown and a reliable background alarm.
#18 Local LLM doesn't have internet access — need to solve this Achievable Already solved

Current state: tools/weather.py (Open-Meteo) and tools/web_search.py (DuckDuckGo) already fetch live data from the internet. The results are injected into the LLM prompt as a LIVE DATA: block via build_tool_augmented_prompt().

Assessment: This is already solved architecturally. The LLM itself doesn't access the internet — the application layer fetches data and feeds it to the model as context. This is the correct pattern (RAG-style augmentation).

Improvement: The real gap is coverage and reliability: (1) Add more data sources: currency exchange (free API from exchangerate-api.com), news headlines (RSS feeds from BBC/Al Jazeera — no API key needed), prayer times (aladhan.com API — relevant for Egyptian users). (2) Add a data freshness indicator in the prompt: "Weather data fetched 2 seconds ago" vs "Search results cached 5 minutes ago" so the model can hedge appropriately. (3) Add a tool failure fallback: if DuckDuckGo is unreachable, try a cached result or tell the user "I couldn't reach the internet, but here's what I know from my training data."
